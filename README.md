# wordstat_analyzer_web_service

# 📂 Структура проекта Wordstat API

### 1. `app/api/` — Уровень обработки запросов
* **Что внутри:** Файлы-роутеры (например, `wordstat.py`, `auth.py`).
* **Зачем:** Здесь описываются эндпоинты (URL), принимаются данные от пользователя и возвращаются ответы. Здесь нет сложной логики БД, только вызовы сервисов и CRUD.

### 2. `app/services/` — Бизнес-логика и интеграции
* **Что внутри:** Логика взаимодействия с внешними API (Яндекс Вордстат).
* **Зачем:** Если завтра Яндекс изменит формат своего API, мы поправим код только здесь. API вызывает Service -> Service получает данные извне -> Передает их в CRUD для сохранения.

### 3. `app/crud/` — Create, Read, Update, Delete
* **Что внутри:** Прямые асинхронные запросы к базе данных.
* **Зачем:** Вся работа с SQLAlchemy. Здесь обрабатываем сложные связи, такие как генерация `group_id` для объединения нескольких фраз в один запрос.

### 4. `app/schemas/` — Валидация данных
* **Что внутри:** Схемы Pydantic.
* **Зачем:** Гарантируют, что на вход пришел правильный JSON.

### 5. `app/core/` — Глобальные настройки
* **`config.py`:** Чтение переменных окружения (`.env`), ключи API, настройки БД.
* **`database.py`:** Создание движка (Engine) и генератора сессий для БД.
* **`security.py`:** Хеширование паролей и создание/проверка JWT-токенов.

---

## 🚀 Вспомогательные файлы в корне
* **`main.py`:** Точка входа. Собирает все роутеры из `app/api/` в одно приложение.
* **`alembic/`:** Миграции базы данных. Позволяют изменять структуру таблиц без потери данных.
* **`docker-compose.yml`:** Описывает контейнеры для быстрого запуска.
* **`models.py`:** Описание базы данных. SQLAlchemy модели.
  
---

## 🔄 Поток данных (Data Flow)
1. **User** -> делает POST запрос на `/wordstat/search`.
2. **API** -> проверяет токен через `get_current_user`.
3. **Service** -> отправляет запрос в Яндекс и получает JSON.
4. **CRUD** -> создает запись в `top_requests` и пачку записей в `top_request_items`.
5. **API** -> возвращает успешный статус пользователю.

# 📊 Состояние наполнения БД на этом этапе

На текущий момент при вызове эндпоинта `/search` данные распределяются по трем основным таблицам

## 1. Таблица `search_phrases` (Справочник фраз)
Это реестр всех уникальных запросов, которые когда-либо проходили через систему.
* **Что записывается:**
    * `phrase`: Текст поискового запроса.
    * `user_id`: ID пользователя, который первым (или текущим) добавил эту фразу.
    * `created_at`: Дата и время первого появления фразы в системе.
* **Логика:** Система сначала ищет фразу в этой таблице. Если фраза найдена, используется её существующий `ID`. Если нет — создается новая запись.

## 2. Таблица `top_requests` (Лог поисковых сессий)
Каждый раз, когда нажимается кнопка «Поиск», здесь создается «паспорт» этого события.
* **Что записывается:**
    * **`group_id`**: Уникальный числовой идентификатор группы (генерируется на основе времени). Позволяет объединить несколько фраз в один логический запрос.
    * `search_phrase_id`: Внешний ключ, указывающий на ID из таблицы `search_phrases`.
    * `user_id`: Кто именно совершил этот поиск.
    * `total_count`: Общее количество показов основной фразы по данным Яндекса.
    * `requested_at`: Точное время запроса.

## 3. Таблица `top_request_items` (Результаты выдачи)
Сюда попадает содержимое массива `topRequests` из ответа API (левая колонка Вордстата).
* **Что записывается:**
    * `top_request_id`: Связь с родительской записью в `top_requests`.
    * **`phrase`**: Текст похожей или вложенной фразы из результатов выдачи.
    * **`count`**: Количество показов для этой конкретной под-фразы.
* **Объем:** Обычно создается **50 записей** на один поисковый запрос.
